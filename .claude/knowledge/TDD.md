# TDD

Test-Driven Development 가이드입니다.

## 기본 사이클

```
Red → Green → Refactor
 │      │        │
 │      │        └── 코드 개선 (테스트 유지)
 │      └── 최소 구현 (테스트 통과)
 └── 실패하는 테스트 작성
```

## 상세 절차

### 1. Red (실패하는 테스트)

```
// 먼저 테스트 작성
test("기능 설명", () => {
    // 테스트 코드
});
```

- 컴파일 에러도 "Red" 상태
- 테스트가 실패하는 이유가 명확해야 함

### 2. Green (최소 구현)

```
// 테스트를 통과시키는 최소 코드
```

- "최소"가 핵심 - 하드코딩도 OK
- 완벽한 코드가 아니어도 됨

### 3. Refactor (개선)

- 중복 제거
- 명확한 이름
- 구조 개선
- **테스트는 계속 통과해야 함**

## 핵심 규칙

1. 테스트 없이 프로덕션 코드 작성 금지
2. 한 번에 하나의 테스트만 작성
3. 버그 수정은 재현 테스트부터
4. 리팩토링 중 새 기능 추가 금지

## 안티 패턴

- 테스트 먼저 안 쓰고 나중에 추가
- 여러 테스트 한꺼번에 작성
- Green 단계에서 과도한 구현
- 리팩토링 건너뛰기
